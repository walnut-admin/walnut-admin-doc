# Shared 文档编写规范

本文档定义了 `src/zh-CN/content/shared/` 目录下技术文档的统一编写规范。

## 文档结构

所有共享文档应遵循以下标准结构：

```markdown
# <WPageTitle></WPageTitle>

:::warning
- 由claude生成
:::

## 介绍

[核心概念和背景说明]

[设计思想的核心要点，使用编号列表]

[参考来源或设计理念说明]

[重要说明事项]

## 相关链接

- [体验地址](https://www.walnut-admin.com/)
- [相关技术的官方文档/RFC/仓库]（如果有）

## 关键影响维度说明（可选）

[对系统有重大影响的关键参数或配置]

### 1. 参数名称

[参数说明]

- 具体特性
- 注意事项

## 流程图

:::tabs
== 流程名称 1
```mermaid
[Mermaid 流程图代码]
```

== 流程名称 2
```mermaid
[Mermaid 流程图代码]
```
:::

## 与其他系统的协作（可选）

[说明该功能如何与其他系统模块协作]

## 补充说明

### 安全特性

1. **特性名称** - 说明

### 存储策略（可选）

[表格形式说明存储位置、内容、有效期、用途]

### 性能优化

1. **优化点** - 说明

### 局限性

1. **局限点** - 说明
```

## 详细规范

### 1. 页面标题

- 使用 `<WPageTitle></WPageTitle>` 组件自动生成标题
- 不需要手动添加一级标题

### 2. Warning 块

- 位于文档开头，说明内容来源或生成方式
- 标准格式：
  ```markdown
  :::warning
  - 内容由免费的闪电说提供语音识别，再由 gpt 提供的去口头化润色（可选）
  - 流程图由 Claude 生成
  :::
  ```
- 如果内容完全手写，可以只保留流程图说明

### 3. 介绍部分

**结构要求：**
1. 第一段：简要说明技术背景和应用场景
2. 核心思想：使用编号列表（1-5 点）说明关键设计思路
3. 参考来源：说明设计参考或技术来源（如果有）
4. 重要说明：使用无序列表说明注意事项、适用场景、局限性等

**示例：**
```markdown
## 介绍

在现代 Web 应用中，[技术名称] 是 [应用场景]。Walnut Admin 实现了一套基于 [核心技术] 的 [功能描述]。

这套机制的核心思想是：

1. **关键点 1**：说明
2. **关键点 2**：说明
3. **关键点 3**：说明

需要说明的是：

- 适用场景说明
- 安全性说明
- 局限性说明
```

### 4. 相关链接

**必须包含：**
- 体验地址（固定链接）

**可选包含：**
- 官方文档/RFC 规范
- 开源库/仓库地址
- 参考实现

**不应包含：**
- 前端代码仓库链接
- 后端代码仓库链接

**格式要求：**
```markdown
## 相关链接

- [体验地址](https://www.walnut-admin.com/)
- [技术官网](链接)
- [RFC 规范](链接)
- [开源库](链接)
```

### 5. 关键影响维度说明（可选）

- 用于说明对系统有重大影响的参数或配置
- 每个维度使用三级标题
- 包含参数说明、特性列表、注意事项

**示例：**
```markdown
## 关键影响维度说明

### 1. 参数名称

`参数名称` 是 [简要说明]：

- 特性 1
- 特性 2
- 注意事项
```

### 6. 流程图

**要求：**
- 使用 VitePress tabs 插件包裹多个流程图
- 使用 Mermaid 语法绘制
- 流程图类型：sequenceDiagram（时序图）
- 参与者命名：使用中文 + `<br/>` 换行 + 英文注释

**格式：**
```markdown
## 流程图

:::tabs
== 流程名称 1
\`\`\`mermaid
sequenceDiagram
    participant Client as 客户端<br/>(Frontend)
    participant Server as 服务器<br/>(Backend)

    Note over Client,Server: 步骤说明
    Client->>Server: 请求
    Server-->>Client: 响应
\`\`\`

== 流程名称 2
\`\`\`mermaid
[另一个流程图]
\`\`\`
:::
```

**Mermaid 规范：**
- 使用 `Note over` 标注步骤或阶段
- 使用 `activate`/`deactivate` 标注激活状态
- 使用 `alt`/`else`/`end` 表示条件分支
- 使用 `opt`/`end` 表示可选步骤
- 箭头类型：
  - `->>` 实线箭头（请求）
  - `-->>` 虚线箭头（响应）

### 7. 与其他系统的协作（可选）

- 说明该功能如何与其他系统模块集成
- 使用三级标题区分不同系统
- 可以添加内部文档链接

**示例：**
```markdown
## 与其他系统的协作

### 功能 A 在系统 B 中的作用

[说明集成方式和作用]

详细的 [系统 B] 流程请参考 [链接](/content/shared/xxx)。
```

### 8. 补充说明

**标准子章节：**
1. **安全特性** - 列举安全相关的设计点
2. **存储策略**（可选）- 使用表格说明存储位置、内容、有效期、用途
3. **性能优化** - 列举性能优化措施
4. **局限性** - 说明已知的限制和不足

**格式：**
```markdown
## 补充说明

### 安全特性

1. **特性名称** - 说明
2. **特性名称** - 说明

### 存储策略

| 存储位置 | 内容 | 有效期 | 用途 |
|---------|------|-------|------|
| Cookie | 数据 | 30天 | 说明 |

### 性能优化

1. **优化点** - 说明
2. **优化点** - 说明

### 局限性

1. **局限点** - 说明
2. **局限点** - 说明
```

## 写作风格

### 语言规范

- 使用简体中文
- 技术术语使用英文原文 + 中文注释（首次出现时）
- 代码、配置项、文件路径使用反引号包裹
- 避免口语化表达

### 格式规范

- 使用 Markdown 标准语法
- 代码块指定语言类型
- 列表项之间不留空行
- 表格对齐使用管道符

### 内容规范

- 避免过度详细的实现细节（如 Redis 缓存键的完整列表）
- 聚焦于设计思想、流程说明、关键配置
- 补充说明部分简明扼要
- 流程图清晰标注步骤和参与者

- `opaque.md` - OPAQUE 认证（关键影响维度说明）

- `device.md` - 设备追踪（系统协作说明）

## 注意事项

1. **不要包含的内容：**
   - 前后端代码仓库链接（在相关链接中）
   - 过于详细的 Redis 缓存结构
   - 具体的代码实现细节
   - 业务逻辑的详细说明

2. **必须包含的内容：**
   - 体验地址链接
   - 核心设计思想
   - 至少一个 Mermaid 流程图
   - 补充说明（安全特性、性能优化、局限性）

3. **可选包含的内容：**
   - 关键影响维度说明
   - 与其他系统的协作
   - 存储策略表格
   - 外部技术文档链接
